# Build a Nami.Identity.Bff package by composing OSS-permissive pieces rather than adopting a commercial BFF

## Context and Problem Statement

A commercial identity vendor sells a commercial backend-for-frontend (BFF) package for SPAs: server-side tokens, automatic/silent renewal, anti-forgery, a backend proxy, session, and logout. Nami wants parity for consumer SPAs but as OSS. Research on 2026-07-04 confirmed that the commercial BFF package carries a paid production license, so it is rejected under ADR-0026. Nami already uses a BFF *pattern* for the admin app (ADR-0020) and recommends a BFF for SPAs (a non-extractable key does not stop XSS, so the BFF is the real mitigation), but it has no reusable BFF package for consumers. How should the BFF be provided?

## Decision Drivers

* Parity with a commercial BFF, but OSS and free.
* The token must never reach the browser, which is the real XSS mitigation for SPA and mobile-web clients.
* Reuse one BFF for both the admin app and consumer SPAs.
* Give consumers a secure default out of the box.

## Considered Options

* Adopt the commercial BFF package
* Build `Nami.Identity.Bff` by composing OSS-permissive pieces (a reverse proxy, an access-token-management library, and ASP.NET cookie/OIDC/antiforgery)
* Ship no BFF and leave it to the consumer

## Decision Outcome

Chosen option: "Build `Nami.Identity.Bff` by composition", at parity with a commercial BFF but from permissive pieces (several already used elsewhere in Nami), because the commercial package is rejected by the OSS-only policy and leaving it to consumers risks them putting the token in the browser.

* **A. Components (composed from OSS-permissive pieces):**

  | Feature | Built from | License |
  |---|---|---|
  | Server-side token storage plus automatic/silent renewal | built-in ASP.NET Core OIDC token handling plus a server-side refresh component, or an OSS-permissive access-token-management library (ADR-0026), non-vendor-branded | MIT/permissive |
  | Backend API proxy (local and remote) | YARP | MIT |
  | Anti-forgery (CSRF), in two profiles | a server-rendered admin form uses an ASP.NET antiforgery token; a JS/SPA proxy uses a custom header plus strict CORS plus rejecting the CORS-simple content type | MIT |
  | Session | a `__Host-` HttpOnly/Secure/SameSite cookie plus a server-side session (ADR-0003) | MIT |
  | Logout (plus back-channel) | ASP.NET OIDC sign-out plus a back-channel logout receiver (ADR-0019) | MIT |

* **B. Security principles (consistent with ADR-0019/0020/0024):**
  * The token **never** reaches the browser — an httpOnly cookie holds the session, and the access/refresh tokens stay server-side — which is the real XSS mitigation for SPA and mobile-web (a non-extractable key is not enough).
  * Anti-forgery is mandatory on every state-changing call, in two profiles: a server-rendered admin form uses an antiforgery token, and a JS/SPA proxy uses a custom header plus strict CORS plus rejecting the CORS-simple content type (a custom header alone is bypassable by a simple-content-type cross-site form POST, so a content-type guard plus strict CORS are required).
  * The BFF proxy forwards only to an allow-listed backend and attaches the bearer server-side.
* **C. Packaging (ADR-0027):** `Nami.Identity.Bff` (plus a `.Bff.Yarp` for the remote proxy, split as the commercial product splits it) is enabled by a consumer via `services.AddNamiBff()...`. The admin app (ADR-0020) is refactored to use this same package, giving one BFF with two consumers (the admin app and consumer SPAs) and no duplicated code.
* **D. Scope boundary.** This BFF covers browser-SPA and mobile-web clients, where the token stays server-side. Token management for a resource-consuming RP or worker (a service calling downstream APIs with its own client-credentials/refresh tokens) is **out of the batteries-included surface**; integrators add an OSS-permissive token-management library directly, with an optional thin machine-to-machine sample.

### Consequences

* Good, because it is at parity with a commercial BFF but OSS and free (the Apache-2.0 release of ADR-0027), reuses pieces Nami already has, and serves both admin and consumers from one BFF.
* Good, because a consumer SPA gets the correct XSS mitigation out of the box (the token is not in the browser).
* Bad, because there is no OSS drop-in, so Nami must compose and maintain it (a medium-to-large effort), with the anti-forgery, proxy, and session paths needing careful testing.
* Bad, because it depends on the reverse-proxy and token-management library versions, which calls for a contract-regression check when they are bumped (the ADR-0021 thinking).

### Confirmation

* The commercial BFF's feature set is documented, and its license is commercial, so it is rejected under ADR-0026.
* The OSS fill is available: YARP (MIT), an OSS-permissive access-token-management approach (ADR-0026), ASP.NET cookie/OIDC/antiforgery (MIT), and a Microsoft OIDC-BFF reference sample as a pattern.

## Pros and Cons of the Options

### Adopt the commercial BFF package

* Good, because it is a turnkey, well-supported drop-in.
* Bad, because it carries a paid production license and is rejected by the OSS-only policy (ADR-0026).

### Build `Nami.Identity.Bff` by composition (chosen)

* Good, because it reaches parity from permissive pieces (many already used), stays OSS/free, and yields one BFF for both admin and consumers.
* Bad, because it must be composed and maintained rather than dropped in, and the security-sensitive paths need careful testing.

### Ship no BFF and leave it to the consumer

* Good, because it is the least for Nami to build.
* Bad, because it loses developer experience and risks consumers implementing it insecurely, with the token ending up in the browser.
